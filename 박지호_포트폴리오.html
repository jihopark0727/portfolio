04 Measurement · The Turning Point

'선택'이 아니라 '실제 교체'를 셌다

지표를 잘못 잡으면 개선의 방향도 틀어진다. 나는 NSM을 버튼 탭이 아니라 서버에 저장된 실제 교체로 정의했다. 일정이 진짜로 바뀌어야 '무너진 계획 복구'라는 핵심 가치가 전달되기 때문이다.

// North Star Metric NSM = alternative_selected (버튼 탭) → alternative_replaced (실제 교체)
진단

원가의 정체를 찾다

대안찾기·틈새 시간·날씨 대안은 모두 같은 엔진이다. 근처 약 20개 후보를 모아 각각을 GPT로 분류하고, 리뷰·거리로 랭킹해 상위 5곳을 제시한다.

// 추천 엔진 파이프라인 Nearby Search → 20개 후보 → GPT-4o-mini × 20 (mood / space / type 분류) → 리뷰·거리 가중 랭킹 → 상위 5곳 제시

발견: GPT를 20번 돌려도 비용은 거의 오르지 않았다(20콜 ≈ ₩10). 진짜 비용은 20개 장소의 리뷰 상세 조회(Place Details)였다. 원가를 좌우하는 건 AI가 아니라 외부 API 호출 횟수였던 것이다.

조치 1

분류는 값싼 데이터로

20개 후보의 mood/type 분류는 Nearby 응답 필드와 DB 캐시만으로 처리.

조치 2

비싼 호출은 최종 후보만

리뷰 Details는 20곳 전부가 아니라 최종 5곳에만 호출.

조치 3

한 번 분석하면 재사용

mood/type 매핑을 DB에 영속화해 재검색 시 추가 비용 0원.

시나리오리뷰 Details 호출검색 1회 단가
콜드 · 20개 전부 호출20곳≈ ₩600
권장 · 최종 5개만 호출5곳≈ ₩190
웜 · 대부분 DB 캐시0–2곳≈ ₩60

데이터 네트워크 효과. 장소 DB가 채워질수록 웜 비율이 올라가고 검색 단가는 계속 떨어진다. 쌓인 데이터가 그대로 원가 경쟁력이 되는 구조다. 실제 단가는 cost_per_search 이벤트에 details_call_count를 함께 로깅해 추적하도록 설계했다.

※ 위 단가는 실제 API 호출 구조를 바탕으로 산출한 설계 단계의 추정치다(환율 ₩1,380 기준). 서비스는 베타 단계까지 진행했으며, 실매출은 발생하지 않았다.

07 Reflection

배운 것

지표가 개선을 만든다

'실제 교체'를 NSM으로 잡지 않았다면 42%의 누수를 발견하지 못했을 것이다. 무엇을 세느냐가 무엇을 고칠지를 결정한다.

기술은 곧 경험이다

직접 구현해봤기에, 400초의 대기가 왜 '이탈'로 찍히는지 몸으로 알았다. 기획과 코드 사이를 오갈 수 있다는 건 나의 가장 큰 무기였다.

성장과 품질의 우선순위

핵심 가치가 새는 상태에서 유저를 늘리는 건 밑 빠진 독이라 판단해, 마지막 스프린트를 '전환 고도화'에 집중했다. 전환은 증명했지만 — 성장과 품질을 완전히 순차로 두기보다 일부라도 병렬화했어야 했다는 걸 배웠다.

01 Overview

"자리 있을까?"를 감이 아니라 데이터로

카공족·노트북족·모임 등으로 카페 수요는 늘었지만, 빈 좌석을 미리 확인할 방법은 여전히 부족합니다. 자리를 찾아 매장을 한 바퀴 돌거나, 자리가 없어 발길을 돌린 경험은 누구에게나 있죠.

주차(카카오T)·웨이팅(캐치테이블)처럼 혼잡도는 이미 하나의 정보가 되었지만, 카페는 이 흐름에서 소외돼 있었습니다. 기존 서비스는 사용자 후기 기반이라 즉각적인 변화에 대응하기 어렵고 주관이 개입돼 정확도가 떨어졌습니다. Seatify는 이 한계를 AI 이미지 분석으로 보완합니다 — 천장 CCTV 영상을 YOLO로 분석하고 Homography로 평면 투영해, "진짜" 비어 있는 좌석을 실시간으로 보여줍니다.

기획 전환 · Pivot
전환
처음엔 '최애 축구선수 하이라이트 자동 편집'을 만들려 했지만, 잦은 화면 전환과 동일 선수 추적의 한계로 실시간성이 부족했습니다. 팀 논의 끝에 AI로 실생활의 불편을 푸는 방향으로 전환한 결과가 Seatify입니다.

구현

백엔드 · DB · 실시간 처리

3초마다 좌석 상태를 조회하는 실시간 서비스 특성상 쿼리 지연과 추론 지연을 줄이는 것이 핵심이었습니다.

Case 01 · DB 성능 최적화

3초마다 특정 좌석을 조회 → 데이터가 쌓일수록 풀 스캔 비용이 실시간성을 위협. (cafe_id, seat_number) 복합 인덱스를 추가해 인덱스 레인지 스캔으로 전환, EXPLAIN으로 검증했습니다.

-- 10,000건 더미 데이터 기준 Full Scan → Index Scan 0.0044s → 0.00097s (≈4.6×)
Case 02 · 실시간 처리 개선

Cold Start 구조는 매 요청마다 YOLO 모델을 새로 로드 → I/O·초기화 비용이 반복되고 병렬 요청 시 지연. Persistent 모델 구조로 전환해 서버 기동 시 한 번만 로드하고 이후엔 추론만 수행하도록 개선했습니다.

-- 평균 응답 속도 15s → 7s (≈53% 개선)

이 외에도 시간대별 좌석 혼잡도 예측, 타 카페 추천 기능을 백엔드에 추가했고, 클릭 로그 기반 실시간 추천(Kafka 비동기 파이프라인, "현재 이 카페를 N명이 보는 중")의 설계에 참여 — 동시성 충돌을 피하는 lock-free 비동기 흐름을 목표로 했습니다.

04 My Contribution

3인 팀에서 백엔드 전반을 단독으로 담당했습니다. API 서버 설계·구현부터 인증, 파일 처리, DB, AI 연동, PDF 변환까지 서버 사이드 전체를 책임졌고 — 무엇을 만들지는 현장 인터뷰에서 도출한 요구사항이 결정했습니다.

01

RESTful API · 계층형 아키텍처

Controller → Service → Repository로 관심사를 분리하고 DTO/Entity 변환을 일관 적용. 대시보드 API는 countEmotions() · findMostFrequentEmotion() 으로 감정 통계를 집계해 시각화 데이터를 반환.

02

인증 · 파일 업로드

JWT 기반 stateless 인증으로 요청별 신원 검증. Multer로 녹음 파일을 저장하되 타임스탬프를 붙여 파일명 충돌을 방지.

03

감정 분석 파이프라인 연동

STT → 감정 분류 → 요약의 파이프라인을 백엔드에서 단계별로 연동·처리. DB는 7개 테이블로 정규화하고, 감정 결과를 세션·문장 단위로 저장해 파도/레이더 시각화에 재사용.

핵심 판단 · KoBERT → OpenAI API 전환
문제
Naver Clova STT + KoBERT 조합으로 감정 분류를 시도했으나, 긴 문장·복잡한 문맥에서 정확도가 낮고 재학습·튜닝에 과도한 리소스가 필요했습니다.
해결
감정 분석·요약을 OpenAI API로 전환. 별도 학습 없이 즉시 적용 가능하고, 복잡한 문맥에서도 감정 분류와 요약을 동시에 높은 정확도로 수행하도록 개선했습니다.
결과
STT 데이터를 더 신뢰성 있게 분석·활용 → 감정 분석 정확도와 처리 효율을 모두 향상.
'모델이 좋다'가 아니라 '우리 데이터에서 잘 동작하는가' — 검증을 기준으로 삼은 기술 의사결정이었다.
